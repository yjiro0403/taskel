# Taskel AI 機能要件定義書

## 1. はじめに
### 1.1 プロダクト定義
Taskelは単なるタスク管理ツールではなく、**「目標管理とタスクシュート時間管理術を統合したパーソナル時間管理OS」**です。
Manager（管理者）のためではなく、**Executor（実行者）**のために存在し、ユーザーが「Flow State」に入り、実行に没頭できる環境を提供することを目的としています。

### 1.2 TaskelにおけるAIの役割
*   **Assist, not Delegate**: AIはユーザーの代わりに勝手にタスクを処理したり、スケジュールを勝手に最適化したりしません（それはMotion等の役割）（ユーザが許可をしたなら大丈夫）。
*   **Calibration（校正）**: ユーザーの時間感覚（見積もり vs 実績）のズレを指摘し、自己認識を促すパートナーとして機能します。
*   **Bridge（接続）**: 年次・月次目標と、今の瞬間の実行（タスク一回分）を滑らかに接続するサポートを行います。

### 1.3 スコープ
本ドキュメントは、Taskelアプリケーション内に統合される「AIサイドバー」および関連するAI機能の要件を定義します。

---

## 2. システム概要
### 2.1 コンセプト
*   **自然文入力 → AI提案 → ユーザー確認 → 実行**: 
    *   AIはあくまで「提案」を行い、最終決定権は常にユーザーにあります（User Agency）。
    *   「タスク作成カード」を経由することで、ユーザーは自分の意図が正しく反映されるかを確認し、無意識な「タスクの死蔵」を防ぎます。

### 2.2 ターゲットユーザーの課題解決
*   **「時間盲（Time Blindness）」の克服**: 見積もりの甘さや時間の使いすぎをAIが優しく指摘します。
*   **目標の形骸化防止**: 「そのタスクは今月の目標のどれに紐づきますか？」といった問いかけにより、意識の接続をサポートします。

## 3. 機能要件

### 3.1 AIサイドバー（Execution Partner Interface）
*   **チャットUI**: 画面右側（またはモバイルではドロワー）に常駐。
*   **コンテキスト認識**: 現在のビュー（Today, Project, Goalなど）を理解し、その文脈に沿った支援を行う。

### 3.2 自然言語によるタスク作成・編集（入力の摩擦低減）
*   **自然言語解析**:
    *   タイトル、期日、時刻に加え、**「見積もり時間」**の入力を重視する（例：「30分で資料作成」）。
    *   ユーザーが見積もり時間を入力しなかった場合、AIは過去のデータや一般的基準から「見積もり時間」を提案する（「これだと45分くらいかかりそうですがどうですか？」）。
*   **タスク作成カード（確認用UI）**:
    *   **必須項目**: タイトル、日付、**見積もり時間**、リスト/プロジェクト。
    *   **目標項目**: Taskel特有の「Goals」への紐づけフィールドを含める。
    *   AIが生成した内容をカード形式でプレビューし、ユーザーが修正・確定ボタンを押すことでDBに反映する。

### 3.3 計画と整理（Planning & Organizing）
*   **目標からの逆算 (Goal Breakdown)**:
    *   **Yearly/Monthly/Weekly Goal**: ユーザーが目標を入力すると、AIがそれを具体的なタスクに分解して提案する。
        *   例: 「今月の目標『サイト公開』のために、今週やるべきステップはこれですか？」
    *   **Reality Check**: 「今日のセクション見積もり合計が10時間を超えています。あふれたタスクを明日に回しますか？」といった警告を行う。
*   **Inboxの整理**: Inboxにある未分類タスクに対し、適切な「プロジェクト」や「セクション」への振り分けを提案する。

### 3.4 実行支援（Execution & Time Tracking）
*   **タイマー制御**: 自然言語で「今から〇〇開始」と指示すると、タスクを作成し即座にタイマーをスタートさせる（着手のハードルを下げる）。
*   **割り込み対応**: 実行中に割り込みタスクが発生した際、AIに「割り込み」と伝えると、現在のタイマーを一時停止し、新しい割り込みタスクのタイマーを開始するフローを提案する。

### 3.5 継続と成長（Review & Analytics）
*   **日次レビュー (Daily Review)**: 1日の終わりに、完了タスク、未完了タスク、総作業時間、目標達成度を振り返るサマリを生成する。
*   **時間感覚の校正 (Calibration)**: 「見積もり30分に対し、実際は50分かかりました。次は50分で見積もりませんか？」といった適度なフィードバックを行う。
*   **ルーチン提案**: 繰り返されているタスクパターンを検出し、「これをルーチン登録しませんか？」と提案する。

## 4. UI/UX要件

### 4.1 デザイン哲学の継承
*   **Simple, yet Powerful**: ノイズを減らし、実行を阻害しないUI。
*   **Optimistic UI**: AIの応答待ちでユーザーを待たせない。入力中は即座にプレビューを表示する等の工夫。

### 4.2 インタラクション
*   **確認ファースト**: 勝手にデータを書き換えない。「これでいいですか？」というスタンスを崩さない。

## 5. 技術要件

### 5.1 アーキテクチャ
*   **Frontend**: Next.js (App Router), React, Zustand
*   **Backend**: Firebase Functions, Firestore
*   **AI Model**: Gemini 1.5 Pro (JSON mode推奨) - 複雑な文脈理解と高速な応答の両立。

### 5.2 データフロー
1.  User Input -> AI Analysis
2.  **Suggestion Phase**: AI proposes a `TaskCandidate` or `GoalBreakdown` object (JSON).
3.  **Review Phase**: UI renders `TaskCreationCard` or `BreakdownPreview` with the candidate data.
4.  **Action Phase**: User edits/confirms -> Writes to Firestore.
